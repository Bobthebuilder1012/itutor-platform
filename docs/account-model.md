# iTutor Account Model — Phase 2 Reference

> Written against migrations 127–133. Phase 3 (WiPay wiring) must respect all rules here.

---

## 1. Student account types

There are exactly two student origins. The type is stamped at creation and is **immutable forever**.

| Type | `account_type` | `billing_mode` | Created by |
|---|---|---|---|
| Self-registered | `self_registered` | `self_allowed` | Student signs up directly |
| Parent-created | `parent_created` | `parent_required` | Parent creates from their dashboard |

### Immutability rules

- `profiles.account_type` cannot be changed after insert. Enforced by `trg_enforce_student_account_type`.
- A `self_registered` student **can never** be given a parent. Any attempt to `INSERT INTO parent_child_links` where the child's `account_type = 'self_registered'` raises an exception via `trg_enforce_parent_child_link_policy`.
- No API route may accept `parent_id`, `account_type`, or `billing_mode` as mutable fields after account creation.

### Enrollment flows by type

| Who enrols | Student type | Consent required |
|---|---|---|
| Student requests to join | `self_registered` | No parent consent |
| Student requests to join | `parent_created` | `parent_consents` row required |
| Parent enrols directly | `parent_created` | Consent is implicit — record created automatically |
| Tutor invites | Either | Student (or parent for `parent_created`) must accept |

## 2. Parent account

The parent is a **first-class role** (`profiles.role = 'parent'`). It has:

- Its own auth session (Supabase Auth, same as tutor/student).
- `notification_preferences` (same JSONB shape as other roles).
- `parent_settings` (JSONB): `{ notify_feedback, notify_payments, notify_warnings, consent_required_for_enroll }`.
- Suspension fields inherited from `profiles` (is_suspended, suspension_reason, etc.).
- A `parent_child_links` row per child.

Parents **cannot** be students or tutors simultaneously. Role is fixed at registration.

## 3. Class membership status values

After migration 127, `group_members.status` has these values:

| Value | Meaning |
|---|---|
| `invited` | Applied by student/parent OR invited by tutor; awaiting acceptance/approval |
| `active` | Approved and enrolled |
| `suspended` | Access paused (e.g. missed payment + auto_suspend = true) |
| `removed` | Permanently removed (by tutor, parent, or student withdrawal) |

**Old values** (`pending`, `approved`, `denied`) were migrated in 127 and must not appear in new code.

---

## 4. Billing model immutability

`groups.billing_timing` (`start` | `end`, per-month only) is **immutable once set**. Enforced by `trg_billing_timing_immutable`:

- Any UPDATE that changes a non-NULL `billing_timing` raises a `check_violation`.
- Phase 3 (WiPay) must set `billing_timing` on first publish and never expose it as editable after that.
- The Create Class UI already renders `billing_timing` only when `billing_model = 'per_month'`.

---

## 5. Payments architecture

Two payment tables coexist with different responsibilities:

| Table | Purpose | Mutability |
|---|---|---|
| `payments` | WiPay gateway transaction log | Immutable after creation |
| `class_payments` | Per-member billing ledger (due/paid/overdue/waived) | Mutable (status changes) |

`class_payments` rows are generated by the system:
- `per_session` → one row per `group_session_occurrences` row for that member.
- `per_month` → one row per calendar month during active enrollment.
- `prepaid` → one row per block purchase (linked to `prepaid_blocks`).

When a WiPay payment succeeds, its `id` is written to `class_payments.wipay_payment_id`.

---

## 6. Sessions unified model

`sessions.booking_id` is now **nullable**. Exactly one of `booking_id` or `class_id` must be set per row — enforced by `sessions_exactly_one_source` CHECK constraint.

| Source | `booking_id` | `class_id` |
|---|---|---|
| One-time 1:1 (existing flow) | NOT NULL | NULL |
| Recurring 1:1 class | NULL | NOT NULL (student_limit = 1) |
| Group class session | NULL | NOT NULL |

One-time 1:1 bookings continue to use the existing `bookings → sessions` flow unchanged.

---

## 7. Tutor reliability

`tutor_response_metrics.feedback_on_time_rate` tracks the % of `parent_feedback_reports` submitted by their `due_at` in the last 90 days.

Call `public.recalculate_feedback_on_time_rate(tutor_id)` from:
- Cron after the feedback period closes.
- After a report transitions to `sent`.

Read via the `public.tutor_reliability` view (same RLS as `tutor_response_metrics`).
